# Calculating the Composites

Good question to nail down before any data warehouse work begins. Let me work through each metric precisely — source, formula, and the gotchas your data engineering team will need to handle.

---

## AI Engagement Index

**Premium Requests per Developer**

- **Source:** Copilot Usage Metrics API — `user_initiated_interaction_count` per user per day
- **Formula:** `SUM(user_initiated_interaction_count) / COUNT(active_users)` over the reporting period (sprint or month)
- **Gotchas:**
  - This counts all explicit user-initiated interactions — chat, agent mode, inline edit prompts — but excludes passive autocomplete suggestions. That is intentional; it measures deliberate AI engagement.
  - Denominator should be *active* users, not licensed seats. A developer on PTO who shows zero interactions should not dilute the average.
  - The API aggregates at the org level by default. User-level daily data requires iterating the per-user endpoint and joining on `login`.

**Acceptance Rate (Trend Only)**

- **Source:** Copilot Usage Metrics API — `total_code_acceptances` and `total_code_suggestions` per user per day
- **Formula:** `SUM(total_code_acceptances) / SUM(total_code_suggestions)` over the reporting period, calculated at team level
- **Trend calculation:** compute the rate for each reporting period, then report the period-over-period delta (e.g., +2.3 percentage points this month vs. last)
- **Gotchas:**
  - Report the *trend*, not the absolute rate, for two reasons: the healthy range is narrow (25–35%), and absolute rate comparisons across teams with different languages and task types are misleading
  - A team writing mostly boilerplate (config, tests, CRUD) will have structurally higher acceptance rates than a team doing complex algorithm work — that difference reflects task type, not AI fluency
  - Watch for sustained rates above 35%; that is the signal for over-reliance, not high performance

**Active User Rate**

- **Source:** Copilot Usage Metrics API — `total_engaged_users` and `total_active_users`; Copilot Billing API — licensed seat count
- **Formula:** `total_engaged_users / licensed_seats` over the reporting period
- **GitHub's definitions matter here:**
  - `total_active_users` — used Copilot at least once in the period
  - `total_engaged_users` — accepted at least one suggestion; GitHub considers this the more meaningful adoption signal
  - Use `total_engaged_users` as your numerator; it is a higher bar and more honest for executive reporting
- **Gotchas:**
  - Developers on leave, between assignments, or in non-coding roles (scrum masters, some architects) will appear as inactive and suppress the rate. Consider maintaining a "coding population" denominator in your warehouse that excludes non-coding seats.
  - The billing API's `last_activity_at` has a 90-day retention window — useful for identifying persistently dormant seats for license hygiene, but not for weekly engagement tracking.

## Velocity Composite

**Cycle Time**

- **Source:** GitHub commit API (first commit timestamp) + Rally API (story closed timestamp)
- **Formula:** `story_closed_at (Rally) - first_commit_at (GitHub)` per story, then `MEDIAN()` across all stories closed in the reporting period
- **Why median, not mean:** a handful of stories stuck in UAT or blocked on dependencies will skew a mean significantly; median gives a more representative picture of typical flow
- **Gotchas:**
  - This is the hardest metric to instrument cleanly given your variable branching strategy. It requires a reliable join key between GitHub commits and Rally stories. The most common approach is parsing story IDs from commit messages (e.g., `US12345`) or branch names. You will need to assess how consistently your teams include story references — and potentially run a data quality audit before treating this metric as reliable.
  - If story ID tagging is inconsistent, a fallback is `branch_created_at` to `story_closed_at`, which requires mapping branch names to stories — messier but sometimes more available.
  - For teams that commit directly to main (trunk-based), `first_commit_at` is the timestamp of the first commit touching files associated with the story, which requires the story ID in the commit message to identify.
  - Exclude stories closed without any associated commits (pure process stories, documentation) from the denominator to avoid artificially deflating cycle time.

**Story Throughput Rate**

- **Source:** Rally API — stories with `ScheduleState = Accepted` or `Closed` (confirm your org's terminal state) within the reporting period, by team
- **Formula (raw):** `COUNT(stories_closed) / COUNT(developers_on_team)` per sprint or month
- **Formula (points-weighted):** `SUM(story_points_closed) / COUNT(developers_on_team)` per sprint or month
- **Recommendation:** report both. Raw count is intuitive for executives; points-weighted is more honest for technical audiences because it accounts for story size variation.
- **Gotchas:**
  - Story point inflation is a real risk if throughput becomes a watched metric. Establish a baseline before surfacing this to leadership, and track average story size alongside throughput to detect inflation.
  - Rally's `Accepted` vs. `Closed` distinction matters — confirm with your teams which state represents genuine done-done, not just code complete.
  - Developers attributed to multiple teams in Rally will double-count in per-developer calculations. You will need a primary team assignment in your warehouse.

**Commit Count**

- **Source:** GitHub commits API — commits per author per repository per reporting period
- **Formula:** `COUNT(commits) / COUNT(developers)` per team per sprint or month — *per-developer normalized*, not raw org total
- **Gotchas:**
  - Merge commits should be excluded (they inflate counts for teams using merge-based workflows). Filter on `parents.length == 1` to isolate non-merge commits.
  - Bot commits (dependency update bots, CI bots) must be filtered by author login against a known bot list in your warehouse — otherwise teams with active automation look artificially more productive.
  - As noted earlier, treat this as a relative trend signal only. A 15% increase in normalized commit count is a directional indicator; it is not a productivity proof point on its own.

## Bringing it together — the composite calculation

Neither composite produces a single blended score, and that is intentional. A blended index number is easier to game and harder to explain when challenged. Instead, report each metric individually within its composite panel, with trend arrows showing period-over-period direction. The *correlation* between the two composite panels — AI engagement trending up while velocity metrics trend up — is the narrative, not a formula.

The one exception is the executive summary row at the top of the CIO dashboard, where you might want a simplified status indicator per composite. A straightforward approach is a **traffic light based on the majority direction of component metrics**: if two of three metrics in a composite are trending in the right direction, the composite shows green. That is transparent, auditable, and resistant to single-metric manipulation.
