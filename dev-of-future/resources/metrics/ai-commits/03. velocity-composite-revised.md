# Revised Velocity Composite

That is the right tension to name. The velocity composite measures delivery performance, but it does not isolate AI's contribution to it. You need a connective metric that bridges the two. Let me think through your two candidates before recommending one.

**AI Lines of Code Added/Deleted**

You are already capturing `loc_added_sum` from the Copilot API, so this is available. The problem is the one we discussed in the original report — LoC is additive by nature. Copilot suggests additions; it rarely suggests deletions or refactors. So `loc_added` will trend upward almost regardless of quality or productivity, which makes it a weak signal. `loc_deleted` from agent mode is more interesting — it may indicate AI-assisted cleanup or refactoring — but agent mode adoption is probably not widespread enough yet in your org to make it meaningful at scale.

**Premium Request Count**

This is actually the stronger candidate of the two, with an important caveat. Premium requests reflect deliberate, higher-cognitive-effort AI interactions — chat, agent mode, inline editing — as opposed to passive autocomplete. A developer who is actively prompting Copilot for non-trivial tasks is using it in a fundamentally different way than one who is just accepting tab completions. The metric is available from the new Usage Metrics API via `user_initiated_interaction_count`.

The caveat: premium request count can also be gamed or misread. Volume alone does not indicate quality of interaction. But as a *relative* signal — trending upward over time, correlated with velocity improvements — it has more informational content than LoC added.

**A better framing for the composite**

Rather than adding a third AI-specific metric to the velocity composite, I would suggest thinking about this as two linked composites presented together on the dashboard:

```
AI Engagement Index          →    Velocity Composite
─────────────────────────────────────────────────────
Premium requests / developer      Cycle time
Acceptance rate (trend only)      Story throughput rate
Active user rate                  Commit count
```

The left side shows *how deeply developers are using AI*. The right side shows *whether delivery is improving*. Presenting them side by side — with a correlation narrative in the executive summary — is more honest and more defensible than embedding an AI metric inside the velocity composite itself. It also sets up a natural question for leadership: "As AI engagement increases, do we see a corresponding improvement in velocity?" That is a story you can tell over rolling quarters.

The reason I would hold back from embedding an AI metric directly in the velocity composite is Goodhart's Law pressure. The moment premium request count or LoC added appears inside a productivity composite reported to a CIO, some teams will find ways to inflate it. Keeping the two composites linked but separate preserves the integrity of both.

That said — if leadership specifically wants a single blended metric, the most defensible option would be **story throughput per active Copilot user**, which normalizes delivery output against confirmed AI engagement rather than just licensed seats. It is simple, auditable, and tells a clean ROI story.
