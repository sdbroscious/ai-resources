# Assessment of the proposed AI team metric

## Verdict

This is a solid **adoption metric**, but not yet a strong **value metric**.

The structure is good because it separates **breadth** from **depth** instead of collapsing “AI adoption” into one number. That is consistent with current guidance from GitHub Copilot and Microsoft Copilot reporting, both of which distinguish adoption-style metrics from deeper usage and engagement patterns. GitHub’s Copilot metrics documentation explicitly groups metrics into categories such as **adoption** and **engagement**, and its metrics APIs include counts for both **active users** and **engaged users**. Microsoft’s Copilot reporting also tracks **enabled users**, **active users**, and **active user rate**. The broader developer-productivity literature also argues against relying on a single metric alone. The SPACE framework specifically says developer productivity should be measured across multiple dimensions, not reduced to one number.

## What is good about the metric

### 1. Breadth is well framed

**Active User Ratio = active users / total team** is a good starting point for adoption.

That directly answers whether the team is actually participating, and it lines up with how major enterprise AI products are measured today. Microsoft’s Copilot usage reporting defines active users as enabled users who tried Copilot during the selected period, and active user rate as active users divided by enabled users. GitHub also treats daily or periodic active users as a core adoption measure.

### 2. Depth is directionally right

Using a separate **Depth** metric is smart. It avoids the trap of saying a team is “AI-enabled” just because a lot of people tried the tool once.

The instinct behind **Premium Request Ratio** is valid: repeated usage can indicate that the tool is entering real workflow patterns rather than remaining a curiosity.

### 3. Team-level saturation is useful for leadership

**AI Team Saturation Ratio = AI teams / total teams** is a useful executive rollup.

Leaders usually need to know not just whether individuals have sampled the tool, but how many teams have crossed a meaningful threshold of adoption. GitHub’s enterprise rollout guidance similarly emphasizes organization- and team-level measurement when evaluating Copilot success.

## Where the metric is weak

### 1. “Premium requests” is a weak standalone proxy for depth

This is the biggest problem.

More requests do **not necessarily** mean more value. They can also mean:
- people are flailing,
- prompting inefficiently,
- solving unusually hard tasks,
- or using the tool as a search box rather than improving delivery outcomes.

In other words, request volume is an **activity** measure, not necessarily a **success** measure. The SPACE framework is useful here because it distinguishes activity from performance, satisfaction, collaboration, and flow. A team can have high AI activity without meaningful productivity improvement.

### 2. Dividing by active days may distort the signal

**Total team premium requests / total team active days** can reward frequent lightweight use and penalize bursty but valuable usage.

Example:
- Team A uses AI heavily during design spikes, incident response, or refactoring windows.
- Team B sends a few shallow prompts every day.

Your ratio may rate Team B as “deeper” even if Team A is getting more actual value.

### 3. The binary threshold is too blunt

Saying a team is an **AI Team** only if it meets both breadth and depth criteria is easy to explain, but it throws away information.

A team that is just under the depth threshold may be much closer to strong adoption than a team with near-zero usage, yet both become “not an AI team.” For management purposes, a continuous score or maturity band is usually more useful than a hard pass/fail cut.

### 4. It measures adoption, not impact

This metric tells you whether teams are using AI. It does **not** tell you whether AI is helping the organization.

That matters because mature measurement should connect usage to outcomes. DORA’s research remains a good companion here because it focuses on delivery outcomes such as deployment frequency, lead time for changes, and change failure rate. Even if you do not use DORA directly for every IT team, the principle still applies: pair usage metrics with outcome metrics.

## My overall judgment

This metric is good for **rollout governance** and **adoption tracking**.

It is not sufficient for demonstrating **organizational value**, and it is somewhat vulnerable to gaming because people can raise request counts without increasing useful work.

So I would keep the general shape but strengthen it.

## Recommended improvements

### Keep breadth, but tighten the denominator

Use:

**Active User Ratio = active users / eligible users**

That is usually better than dividing by total team if some people are not licensed, are not expected to use the tool, or sit in roles where the tool is irrelevant.

### Replace or supplement depth

Instead of relying mainly on:

**Premium Request Ratio = total premium requests / total team active days**

use a combination such as:

- **Engaged User Ratio** = engaged users / active users
- **Sustained Usage Ratio** = users active on 3+ distinct days in the period / active users
- **Median requests per active user**
- **High-value workflow usage** = percentage of users invoking code review, edit, agent, or other higher-value workflows

This is closer to GitHub’s own distinction between **active users** and **engaged users**, and it is harder to game than a pure request-count metric.

### Add one outcome metric

Pick one or two outcomes relevant to the team type.

For IT or engineering teams, examples include:
- ticket resolution time,
- tickets resolved per analyst,
- mean time to first response,
- PR cycle time,
- lead time for changes,
- change failure rate.

The exact outcome metric can vary by function, but the important thing is that usage alone should not be the finish line.

### Replace binary classification with a maturity model

Instead of:
- AI Team = meets breadth and depth threshold

consider:
- **Emerging**
- **Adopting**
- **Integrated**
- **High Leverage**

That preserves nuance and creates a better management conversation.

## A stronger version of the metric

## AI Team Adoption Index

### Breadth
**Active User Ratio** = active users / eligible users

### Engagement
**Engaged User Ratio** = engaged users / active users

### Sustained Usage
**Sustained Usage Ratio** = users active on 3+ days / active users

### Workflow Depth
**High-Value Workflow Ratio** = users using advanced or embedded AI workflows / active users

### Outcome
One team-relevant KPI improvement for the period

## Portfolio rollup

**AI Team Saturation Ratio** = teams above composite threshold / total teams

## Final assessment

Your current metric is a good **phase-1 adoption scorecard**.

It is not yet a convincing **phase-2 value scorecard**.

The key improvement is to stop treating request volume as the main proxy for depth. Use breadth, engaged usage, sustained usage, and at least one business or delivery outcome. That will make the measure more defensible, less gameable, and more useful to leadership.
