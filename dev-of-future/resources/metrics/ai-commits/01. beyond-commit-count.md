# Measuring Copilot productivity beyond commit count

Commit count tells leadership almost nothing about developer productivity — and pairing it with raw Copilot metrics like acceptance rate or lines generated won't fix the problem. The research is clear: **no single metric or even pair of metrics captures real productivity**. Organizations that report Copilot value effectively combine at least three measurement dimensions — activity, quality, and developer experience — using frameworks like SPACE and DORA to tell a story executives can trust. For a 350-developer organization, the most defensible approach is building a composite dashboard that pairs Copilot adoption signals with cycle time, code quality indicators, and quarterly developer satisfaction data, framed in business terms like FTE equivalents saved and cost-per-feature trends.

GitHub's Copilot metrics API just reached general availability in February 2026 with significantly expanded data, including PR lifecycle metrics and agent-mode tracking. This timing creates an opportunity to build a measurement framework on the newest API before the legacy endpoints sunset on April 2, 2026. But the data warehouse strategy must account for a critical finding across multiple independent studies: **individual AI productivity gains often fail to translate into organizational delivery improvements** — a paradox documented by Faros AI, DORA, and others that demands quality counter-metrics alongside any speed or volume measures.

---

## Commit count is the metric that won't die — but should

The case against commit count as a productivity signal is overwhelming and long-established. Martin Fowler argued in 2003 that "any true measure of software development productivity must be based on delivered business value," and GitClear's analysis of 2,729 data points found the correlation between actual developer effort and commit count can be **as low as 10%** — meaning commit graphs convey essentially zero information about productivity.

The problems are structural, not fixable with normalization. Commit frequency varies by personal style (atomic committers vs. squash-and-merge developers), project phase (architecture and debugging produce few commits), language and framework, and team workflow conventions. A developer who fixes a critical production outage with a single-line change delivers enormous value with one commit. A developer splitting trivial CSS changes into 20 commits looks ten times more "productive."

AI tools make this worse. GitClear's analysis of **211 million lines of code** across 2020–2024 found that code churn — lines reverted or substantially changed within two weeks — rose from 5.5% in 2020 to 7.9% in 2024, with a **0.98 Pearson correlation** between Copilot prevalence and what they term "mistake code." Copy-pasted code increased 48% while refactored code plummeted from 24.1% to 9.5% of all changes. More commits, more lines, less actual value.

Kent Beck and Gergely Orosz crystallized the deeper issue in their response to McKinsey's productivity framework: most metrics capture only effort and output, while ignoring outcome and impact. Beck wrote bluntly: "You can measure effort. You can vaguely correlate effort to output. If you try to extend that to profit you will sacrifice that profit." Nicole Forsgren, creator of both DORA and SPACE, reinforced this: "Productivity is about more than the individual or the engineering systems; it cannot be measured by a single metric or activity data alone."

For a CIO audience, the framing should be direct: commit count is to developer productivity what keystrokes-per-minute is to writing quality. It measures motion, not progress.

## What the Copilot API actually provides — a full inventory

GitHub's Copilot metrics infrastructure is in active transition. The **legacy Copilot Metrics API** (`/orgs/{org}/copilot/metrics`) sunsets on **April 2, 2026**. The replacement **Copilot Usage Metrics API** reached GA on February 27, 2026, with significantly richer data delivered via NDJSON download links. Any data warehouse integration should target the new API exclusively.

**Code completion metrics** form the core dataset. The API provides `total_code_suggestions` (shown to developer), `total_code_acceptances` (accepted by developer), `total_code_lines_suggested`, and `total_code_lines_accepted`, all broken down by language, editor, and AI model. Acceptance rate — acceptances divided by suggestions — is the most widely discussed Copilot metric and the one GitHub's own research identifies as the best single predictor of perceived productivity. But it carries significant caveats discussed below.

**Chat and agent metrics** capture the shift toward conversational AI usage. The new API tracks `user_initiated_interaction_count` (explicit prompts, excluding UI navigation), broken down by mode: `chat_panel_agent_mode`, `chat_panel_ask_mode`, `chat_panel_edit_mode`, and `chat_panel_custom_mode`. For code completions and chat, the API reports `code_generation_activity_count` (distinct output events) and `code_acceptance_activity_count` (accepted suggestions including apply-to-file, insert-at-cursor, and terminal insertions). Agent edit metrics track lines added and deleted when Copilot writes code autonomously — a critical new signal as organizations move from assisted to autonomous AI workflows.

**Lines of code metrics** in the new API distinguish between `loc_suggested_to_add_sum` (what Copilot proposed), `loc_added_sum` (what actually entered the editor), and `loc_deleted_sum` (lines removed, currently from agent edits). The separation of suggested-vs-added provides a more honest measure than the legacy API's single acceptance figure.

**PR lifecycle metrics** are the most strategically valuable addition to the new API. At the enterprise and organization level, the API now returns `pull_requests.total_created`, `total_reviewed`, `total_merged`, and critically, **`median_minutes_to_merge`** — a direct cycle time metric. It also separately tracks `total_created_by_copilot`, `total_merged_created_by_copilot`, and `median_minutes_to_merge_copilot_authored`, enabling direct comparison of human vs. AI-authored PR velocity. Review suggestion metrics (`total_suggestions`, `total_applied_suggestions`, `total_copilot_suggestions`, `total_copilot_applied_suggestions`) measure AI participation in code review.

**Adoption and seat data** comes from both the Usage Metrics API and the separate Copilot User Management API (`/copilot/billing/seats`). The metrics API provides `total_active_users`, `total_engaged_users`, and per-user boolean flags (`used_agent`, `used_chat`, `used_cli`). The billing API provides `last_activity_at` with 90-day retention — the source of truth for identifying dormant seats. For a 350-seat deployment, the ratio of engaged users to licensed seats is the most basic adoption health check.

**Dimensional breakdowns** available include by IDE (with version tracking via `last_known_ide_version` and `last_known_plugin_version`), by programming language, by AI model (including custom/fine-tuned models), and by feature type. The new API supports user-level daily and 28-day aggregation, with historical data available back to October 2025 at the enterprise level.

**Key limitations** to communicate to your data engineering team: the new API has no per-repository breakdown (the legacy API had this only for PR summaries); there are no team-level endpoints (you must derive team views from user-level data mapped to your org structure); no built-in time-saved estimates exist; telemetry-disabled users are invisible; and organization-level metrics can double-count users who belong to multiple orgs.

## Metric combinations that tell a credible productivity story

The most defensible approach for CIO reporting combines Copilot-native data with GitHub platform metrics across three narrative layers: adoption health, delivery impact, and quality assurance.

**Layer 1: Adoption health signals** answer "Are developers actually using the tool?" Seat utilization rate (engaged users ÷ licensed seats) is the foundation. For 350 developers, a healthy target is **75%+ weekly active usage** within six months of rollout. The DAU/WAU ratio indicates stickiness — above 60% suggests Copilot is integrated into daily workflow rather than used sporadically. Track feature adoption distribution across completions, chat, and agent mode to understand how usage matures. Agent mode adoption percentage is an emerging leading indicator of teams shifting toward higher-leverage AI usage.

**Layer 2: Delivery impact signals** answer "Is the tool making us faster?" This is where commit count enters the picture — not alone, but as one input in a velocity composite. The strongest combination pairs **PRs merged per developer per week** (the metric validated in the largest RCTs) with **median time to merge** (now available directly from the Copilot API) and **deployment frequency** (from your CI/CD pipeline data). Presenting these three together shows whether teams are shipping more code, faster, and more frequently. The Copilot API's separate tracking of Copilot-authored vs. human-authored PR merge times enables a direct comparison slide for executives.

**Layer 3: Quality assurance signals** answer "Are we maintaining quality while going faster?" This layer is non-negotiable for credibility with technical leadership and boards. The essential quality metrics include **change failure rate** (from deployment/incident data), **CI/CD build pass rate trends** (from GitHub Actions API), **code churn rate** (lines reverted within 14 days, derived from commit history), and **PR review cycles** (review requests with "changes requested" conclusions, from the Reviews API). If the delivery layer shows acceleration while the quality layer holds steady or improves, the productivity story is credible. If quality degrades, the acceleration story collapses.

For the CIO slide deck, the most powerful single visualization is a **dual-axis trend chart** showing PRs merged per developer (velocity) alongside change failure rate (quality) over the months before and after Copilot deployment, with a vertical line marking the rollout date. Accenture's RCT provides the benchmark: they measured **+8.69% more PRs per developer**, **+15% higher merge rate**, and **+84% more successful builds** — a story where velocity and quality improved simultaneously.

## What the research actually shows — and where it conflicts

The evidence base for AI coding tool productivity is substantial but contradictory, and an honest CIO report must acknowledge both sides.

**The strongest positive evidence** comes from Cui et al.'s multi-company field experiment (2024) — three separate RCTs with **4,867 developers** at Microsoft, Accenture, and a Fortune 100 company. They found an aggregate **26% increase in completed PRs per week**, with Microsoft seeing 13–22% gains and Accenture 7.5–8.7%. This is the largest and most methodologically rigorous study to date. GitHub's earlier RCT with recruited developers found **55.8% faster task completion**, though with a wide confidence interval (21–89%) on a single standardized JavaScript task.

Ziegler and Kalliamvakou's 2024 study in Communications of the ACM, matching survey responses from 631 developers with IDE telemetry, found that **acceptance rate is the single best predictor of perceived productivity** (p < .0001) — better than persistence metrics or code contribution volume. However, the authors note "considerable unexplained variance," meaning acceptance rate alone explains only a portion of the productivity picture.

**The critical counter-evidence** demands attention. The METR study (2025) ran an RCT with 16 experienced open-source developers on codebases they had contributed to for 5+ years. Despite developers predicting AI would save them 24% of time, **AI actually increased completion time by 19%** on familiar codebases. Faros AI's telemetry across 10,000+ developers found that while individuals completed 21% more tasks and merged 98% more PRs, **organizational delivery metrics stayed flat** — a "productivity paradox" where downstream bottlenecks in review, testing, and deployment absorbed all individual gains.

The **DORA 2024 report** (surveying ~39,000 professionals) found that as AI adoption increased, there was a **1.5% decrease in delivery throughput** and a **7.2% decrease in delivery stability**. The 2025 DORA report confirmed: "AI doesn't fix a team; it amplifies what's already there." BlueOptima's analysis of 218,000+ developers across enterprises found **only ~4% actual productivity gains** — a stark contrast with GitHub's claims.

Perhaps most importantly, NAV IT's longitudinal study of 25 Copilot users and 14 non-users across 26,317 commits found that **Copilot users were already more productive before the tool was introduced**. This selection bias — more productive developers adopt AI tools first — means many before/after comparisons overstate the tool's causal impact.

The honest synthesis for a CIO: **individual speed gains of 8–25% on output metrics are real and replicable**, but they require mature engineering practices (automated testing, strong CI/CD, code review culture) to translate into organizational outcomes. Without these foundations, speed gains shift bottlenecks downstream and can degrade quality. Developer satisfaction improvements of **85–95%** are the most consistent finding across all studies and may be the most reliable indicator of long-term value through retention and engagement.

## Quality metrics that prevent the speed-without-substance trap

The GitHub REST API provides raw data from which quality signals must be derived — there are no pre-aggregated quality scores. A data warehouse team should build these derived metrics:

**PR-based quality indicators** are the richest source. Time to first review (PR `created_at` to earliest review `submitted_at`) measures review responsiveness and identifies bottlenecks. Review cycles (count of `CHANGES_REQUESTED` review states per PR) serve as a rework proxy — declining cycles after Copilot adoption suggest cleaner first submissions. PR size (`additions` + `deletions`) tracks whether AI encourages appropriately small, reviewable changes. Faros AI found AI adoption correlated with **154% larger average PR size**, which contradicts small-batch best practices and should be monitored closely.

**CI/CD quality signals** from the GitHub Actions API provide the most objective quality data. Build pass rate (`conclusion: success` ÷ total workflow runs) tracked as a weekly trend line reveals whether AI-generated code passes automated checks at the same rate as human code. Accenture's +84% improvement in successful builds is the aspirational benchmark. Re-run frequency (elevated `run_attempt` values) indicates flaky tests or intermittent failures that may correlate with AI-generated code quality. CI duration trends reveal whether growing codebases from AI-assisted development are creating longer build times — an early technical debt signal.

**Code churn rate** — the percentage of lines changed or reverted within 14 days of introduction — is the single most important quality counter-metric for AI-assisted development. GitClear's research established this as the strongest signal of AI-induced quality degradation, and it can be derived from commit history in your data warehouse by comparing file-level diffs across a rolling 14-day window. **Track churn rate separately for high-Copilot-usage teams vs. low-usage teams** to detect whether your specific deployment is producing the degradation pattern seen in aggregate studies.

**Bug rate tracking** via GitHub Issues (filtering by "bug" labels) provides a lagging but ultimately decisive quality signal. If bug creation rate rises proportionally with velocity gains, the net productivity gain is zero or negative. Time-to-close for bugs measures resolution speed. The ratio of bugs-to-features in issue creation over time reveals whether AI-assisted velocity is creating a growing maintenance burden.

## How AI productivity metrics get gamed — and what to watch for

Goodhart's Law — "when a measure becomes a target, it ceases to be a good measure" — applies with particular force to AI coding metrics because the tools make gaming nearly effortless.

**Acceptance rate inflation** is the most immediate risk. If leadership celebrates high acceptance rates, developers will accept mediocre suggestions rather than rejecting and writing better code. GitHub's own research shows acceptance rate varies by experience level — junior developers accept more suggestions (31.9%) than seniors (26.2%), yet senior developers produce higher-quality code. A rising acceptance rate may actually signal declining code quality standards. Laura Tacho, CTO of DX, noted in 2025 that "acceptance rate may be on its way out" as a meaningful metric now that basic adoption is established. The healthy range is **25–35%**; rates above 35% warrant investigation for over-reliance, while rates below 20% suggest poor model-language fit.

**Lines-of-code inflation** is structurally baked into Copilot's design. The tool is additive by nature — it suggests new code but cannot suggest refactors, deletions, or architectural simplifications. Celebrating "lines generated by Copilot" incentivizes the exact behavior that GitClear documented: more duplicated code, less refactoring, growing codebases that become harder to maintain. One longitudinal study of Cursor adoption found that velocity gains concentrated in the first two months before **returning to baseline**, while technical debt accumulated persistently.

**The watermelon metric problem** — dashboards that look green while underlying reality is red — is acute in AI metrics. High acceptance rates + high LOC generated + high seat utilization + rising commit counts can all appear green simultaneously while code churn rises, build times grow, and incident rates climb. The antidote is mandatory pairing: every adoption/velocity metric on an executive dashboard must appear alongside its quality counterweight.

**Perverse incentives from adoption mandates** deserve special attention. Requiring minimum Copilot usage percentages creates compliance theater — developers trigger suggestions they don't need to hit targets. Some organizations have responded with guardrails: one Fortune 500 retailer requires noting the AI assistance percentage in PR descriptions and triggers additional review for PRs exceeding 30% AI content. Others institute "Copilot-free Fridays" to prevent skill atrophy, particularly among junior developers who risk developing "automation bias" — uncritical acceptance of AI output that stunts fundamental skill development.

The mitigation strategy is straightforward: **measure five or more metrics simultaneously** across different dimensions, use oppositional metric pairs (speed vs. quality, throughput vs. satisfaction), never rank individuals by any AI metric, and rotate which metrics receive leadership attention to prevent gaming adaptation.

## Composite frameworks that work for executive reporting

Three frameworks have emerged as credible standards for measuring AI-assisted developer productivity, each with distinct strengths for a 350-developer organization.

**SPACE** (Satisfaction, Performance, Activity, Communication, Efficiency) remains the foundational framework, co-created by Nicole Forsgren and researchers at GitHub, Microsoft, and the University of Victoria. Its core rule: **measure across at least three of the five dimensions**, combining instrumented system data with perceptual survey data, at individual, team, and system levels. For Copilot measurement, Satisfaction maps to quarterly developer surveys, Performance to code quality and deployment success metrics, Activity to Copilot usage and PR throughput (explicitly warned against using alone), Communication to code review patterns, and Efficiency to cycle time and flow-state frequency. SPACE provides the theoretical rigor that gives a CIO dashboard credibility.

**DORA metrics** complement SPACE with four battle-tested delivery performance indicators: deployment frequency, lead time for changes, change failure rate, and failed deployment recovery time (with rework rate added in 2024). The 2025 DORA report's critical finding for AI measurement is that **seven foundational practices** — including automated testing, trunk-based development, and strong feedback loops — determine whether AI adoption improves or degrades these metrics. Organizations with mature practices see AI amplify their performance; those without see degradation. For CIO reporting, DORA metrics provide the "are we actually delivering better software?" validation layer.

**DX Core 4** (2024–2025, from Noda, Tacho, and Forsgren) synthesizes DORA and SPACE into four executive-ready dimensions: Speed (diffs per engineer, team-level only), Effectiveness (Developer Experience Index from a 14-factor survey), Quality (change failure rate), and Impact (R&D investment profile — new value vs. maintenance). Its design principle of **oppositional metrics** — speed counterbalanced by quality, throughput by experience — builds gaming resistance into the framework itself. Results from 300+ organizations show 3–12% engineering efficiency increases and 14% increases in time spent on feature development. This is arguably the most CIO-ready framework available.

For a 350-developer organization with a custom data warehouse, the practical implementation combines all three:

- **Monthly CIO dashboard**: 5–6 metrics drawn from DX Core 4 structure — PRs merged per developer trend, median time to merge trend, change failure rate, Developer Experience Index score, Copilot seat utilization and engagement rate, and an R&D investment ratio (feature work vs. maintenance/debt)
- **Financial framing**: At $150K average loaded developer cost, a 5% efficiency gain across 350 developers equals **17.5 FTE equivalents** or approximately **$2.6M in redirected capacity** annually, against a Copilot license cost of roughly $80K/year (at $19/user/month) — a compelling ROI narrative even with conservative gain estimates
- **Quarterly deep-dive**: Full SPACE-dimension assessment with survey data, Copilot feature adoption maturity (completions → chat → agent mode progression), quality trend analysis (churn rate, build pass rates, bug creation rate), and team-level variation analysis to identify where Copilot delivers the most value by language, project type, and team maturity

The strongest executive narrative is not "Copilot generates X lines of code" but rather: "Since deploying Copilot, our median time-to-merge has decreased by Y%, our developers report Z% higher satisfaction, and our change failure rate has held steady — meaning we're shipping faster without sacrificing quality, at an annual license cost that's a fraction of one developer's salary."

## Conclusion

The path from commit count to a credible AI productivity story requires three shifts in measurement philosophy. First, **replace output metrics with outcome metrics** — PRs merged, cycle time, and deployment frequency tell a velocity story that commit count never can. Second, **pair every speed metric with a quality counter-metric** — build pass rates, code churn, and change failure rate prevent the watermelon dashboard problem that plagues AI adoption reporting. Third, **include developer experience data** — satisfaction surveys are the most consistently validated finding across all Copilot research and serve as both a retention indicator and a leading signal of sustainable productivity.

The research evidence is more nuanced than vendor marketing suggests. Real-world enterprise gains cluster around **8–25% on output metrics** like PRs per week, not the 55% from lab studies. Individual gains often don't translate to organizational delivery improvements without mature engineering practices. And code quality degradation is a documented risk that demands monitoring. An honest CIO report that acknowledges these realities while showing genuine, measured improvements — even modest ones — will be far more credible and durable than one built on inflated vanity metrics.

For immediate action: migrate to the new Copilot Usage Metrics API before the April 2, 2026 sunset, establish baselines across all five SPACE dimensions before expanding Copilot adoption further, build the quality counter-metrics pipeline (churn rate, build pass rates, review cycles) in your data warehouse alongside Copilot data, and frame every executive report in business terms — FTE equivalents, cost per feature, and time to market — rather than raw Copilot telemetry that means nothing to a CIO.
