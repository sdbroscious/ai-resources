This is a straightforward conceptual question — no research needed, and no clarification required. Here is what it means:

A **floor estimate** is a measurement that you know, by design, can only be *lower* than the true value. It establishes a minimum — a baseline below which the real number almost certainly does not fall — but it cannot tell you how much higher the real number actually is.

In the context of AI code contribution:

- The metric counts only what is directly observable and attributable to a single tool (Copilot), through a single mechanism (built-in acceptance actions), in a single lifecycle stage (editor insertion).
- Every known gap — other AI tools, manual copy-paste, AI-influenced-but-rewritten code, agent-mode undercounting — pushes the *true* AI contribution **above** the measured number, never below it.
- Therefore, if the metric says 18% of your new code is AI-generated, the honest interpretation is: *"At least 18% of our new code has AI involvement. The actual figure is likely higher, and we cannot currently quantify by how much."*

The practical reason to frame it this way for leadership is twofold:

- It is **intellectually defensible** — you are not overclaiming precision you don't have.
- It **reframes trend movement** constructively: if the number rises from 18% to 24% over two quarters, that increase could reflect genuine growth in AI usage, improved instrumentation, or both — and either interpretation supports continued investment.

The opposite framing — presenting it as a precise measurement — invites the question "why don't we trust this number?" A floor estimate framing gets ahead of that question.
