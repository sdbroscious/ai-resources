# How to Gather Data for the 5 Headline Measures

You gather those five measures by instrumenting three places at once: the **IDE**, the **code host/CI system**, and a lightweight **analysis pipeline** that joins the events together. If you only measure one layer, the results will be misleading.

Here is the practical setup for each measure.

## 1. AI-originated share of merged diffs

**What you need**  
You need to know which added or changed code in a merged PR first came from an AI suggestion or generation event.

**How to collect it**  
Instrument the IDE or AI coding tool to log events like:

- suggestion shown
- suggestion accepted
- prompt-to-code insertion
- inline generation accepted
- agent-generated file or patch inserted

For each event, capture:

- developer ID
- repo/branch
- file path
- timestamp
- inserted text length
- inserted text hash
- tool/model name
- event type

Then, at merge time, compare the final merged diff against those accepted insertions. You can do this by matching:

- exact text spans
- fuzzy similarity
- token overlap
- AST subtree similarity for code blocks

**Output**  
For each merged PR:

- total added lines
- total changed lines
- AI-originated added/changed lines
- percentage AI-originated

**Important**  
Do not rely only on “accepted suggestions” from a vendor dashboard. That overstates AI contribution because accepted code may later be deleted or heavily rewritten.

---

## 2. Manual keystrokes per merged PR

**What you need**  
A count of actual human typing in source files, not just editor activity.

**How to collect it**  
From the IDE, log:

- character insertions typed by keyboard
- deletions typed by keyboard
- paste events
- autocomplete acceptances
- AI suggestion acceptances

Then classify editor changes into:

- manual typing
- paste
- AI insertion
- IDE autocomplete/snippet insertion

For your metric, count only **manual typing** in code files. You may also want a second metric that excludes comments and markdown.

Then join that with PR metadata:

- PR ID
- merged timestamp
- files changed
- lines added/deleted

**Output**  
For each merged PR:

- total manual keystrokes
- manual keystrokes in source files only
- manual keystrokes per 100 merged LOC
- manual keystrokes per PR
- manual keystrokes per story/issue if you can map PRs to tickets

**Important**  
Do not use git diff as a proxy for typing. A pasted 200-line block and 200 typed characters are very different.

---

## 3. Post-acceptance human edit distance on AI code

This is the most important “reality check” metric.

**What you need**  
You need to know how much AI-inserted code survived intact versus how much humans had to rework it.

**How to collect it**  
When AI-generated code is inserted, create a tracked span or block record with:

- file path
- insertion timestamp
- inserted content
- content hash
- developer ID
- tool/model

Then observe what happens to that span over time until merge:

- unchanged
- lightly edited
- heavily edited
- deleted

You can compute:

- Levenshtein edit distance
- token-level edit distance
- AST edit distance for structured languages

A useful simple version is:

- **survival rate** = proportion of AI-inserted tokens still present at merge
- **human rewrite rate** = proportion of AI-inserted tokens changed by a human before merge

**Output**  
For each AI insertion:

- accepted as-is
- minor edit
- major edit
- fully discarded

Then aggregate by repo, language, model, engineer, and task type.

**Important**  
This measure prevents false optimism. High AI usage with high rewrite rate means AI is drafting, not replacing typing effectively.

---

## 4. Cycle time per shipped feature

This one sits outside the IDE.

**What you need**  
A clean definition of “feature” and a way to map work items to code changes.

**How to collect it**  
From your issue tracker and code host:

- ticket created timestamp
- work started timestamp
- first commit timestamp
- PR opened timestamp
- PR merged timestamp
- production deploy timestamp

Pick a consistent cycle-time definition. Two good choices:

- **coding cycle time**: first commit to merge
- **delivery cycle time**: work started to production deploy

Then segment by AI intensity. For example:

- low AI: <10% AI-originated diff
- medium AI: 10–40%
- high AI: >40%

**Output**  
Compare cycle time across those buckets and over time.

**Important**  
Do not compare raw averages across very different work types. Normalize by:

- bugfix vs feature vs refactor
- repo/service
- team
- complexity band
- language

Otherwise you will confuse easier AI-heavy tasks with true acceleration.

---

## 5. Defect rate for AI-heavy vs human-heavy changes

You need a consistent way to define “defect.”

**How to collect it**  
Combine:

- incident tickets
- bug tickets filed after release
- rollbacks
- hotfix PRs
- test failures in CI after merge
- escaped defects found in QA or production

Map defects back to the originating PR or feature. Then attach the AI-originated share of that change.

**Useful versions**

- bugs per 1,000 merged LOC
- bugs per PR
- rollback rate per 100 deploys
- hotfix rate for AI-heavy vs human-heavy PRs
- incident rate weighted by severity

**Output**  
A comparison like:

- AI-heavy changes: defect rate X
- human-heavy changes: defect rate Y
- mixed changes: defect rate Z

**Important**  
Separate trivial defects from serious ones. A typo and a production outage should not count the same.

# What to instrument, concretely

## A. IDE telemetry

This is where most of the real signal comes from.

Capture:

- keystrokes
- paste events
- suggestion display/accept/reject
- prompt submission
- generated block insertion
- file/language context
- timestamps

You do not need to store raw source text forever. A privacy-conscious design can store:

- hashes
- lengths
- token counts
- diffs
- classification labels

That is often enough for analysis.

## B. Git hosting / PR system

Capture:

- commits
- PR open/merge times
- reviewers
- review comments
- final merged diff
- linked issue/ticket
- deploy linkage if available

## C. CI/CD and quality systems

Capture:

- test failures
- static analysis results
- rollbacks
- hotfixes
- incident links
- post-release bugs

## D. Issue tracker

Capture:

- work item type
- priority
- complexity estimate
- start/end timestamps
- production release timestamp

# A minimal data model

You do not need a giant platform to start. A simple warehouse with these tables is enough:

- **editor_events**
  - developer_id
  - timestamp
  - repo
  - file
  - event_type
  - chars_count
  - tool_name
  - model_name
  - block_id

- **ai_insertions**
  - block_id
  - developer_id
  - file
  - inserted_at
  - inserted_hash
  - token_count

- **prs**
  - pr_id
  - repo
  - author
  - opened_at
  - merged_at
  - linked_ticket

- **pr_files / diffs**
  - pr_id
  - file
  - added_lines
  - deleted_lines
  - final_content_hashes

- **tickets**
  - ticket_id
  - type
  - created_at
  - started_at
  - deployed_at
  - complexity

- **defects**
  - defect_id
  - linked_pr_id
  - severity
  - found_at
  - type

That is enough to compute the five headline measures.

# Easiest way to start without building everything

Do this in phases.

## Phase 1: coarse but fast

Use existing vendor and platform data:

- Copilot/Cursor/Codeium usage dashboards
- PR diffs from GitHub/GitLab
- issue tracker timestamps
- bug/hotfix data

This gives rough estimates for:

- AI-originated share
- cycle time
- defect comparisons

It will not give you reliable manual keystrokes or rewrite distance yet.

## Phase 2: proper IDE instrumentation

Build or deploy an internal IDE extension that logs:

- manual typing
- AI insertions
- paste
- edits to accepted AI blocks

This unlocks:

- manual keystrokes per PR
- post-acceptance edit distance

## Phase 3: normalized analysis

Add segmentation by:

- language
- team
- work type
- repo
- engineer tenure
- model/tool
- task size

This is where your findings become decision-grade instead of anecdotal.

# Practical shortcuts for each metric

If you want a simpler first-pass version:

**AI-originated share of merged diffs**

- Start with accepted AI insertions that survive to merge by fuzzy text matching.

**Manual keystrokes per merged PR**

- Count only direct keyboard text insertion events in source files.

**Post-acceptance human edit distance**

- Measure percentage of accepted AI text still present at merge.

**Cycle time per shipped feature**

- Use first commit to merge, segmented by AI-originated diff share.

**Defect rate**

- Use hotfix PRs and bug tickets within 14 or 30 days of release.

These are not perfect, but they are directionally useful.

# Biggest implementation risks

**Privacy and trust**  
Developers will resist surveillance if you do this badly. Be explicit that you are measuring workflow patterns, not grading individual people. Aggregate by team first unless there is a strong reason not to.

**Bad attribution**  
Copy/paste from a browser LLM, snippets, and pair programming can all look similar unless you classify events carefully.

**Metric gaming**  
Once people know what is measured, they may change behavior. That does not make measurement useless, but it means you should watch for Goodhart’s law.

**False precision**  
You will not get a perfect boundary between human and AI authorship. Aim for useful confidence bands, not fake certainty.

---

# My blunt recommendation

Start with a **90-day pilot** on one or two teams. Instrument only enough to answer these questions:

1. What share of merged diffs appears AI-originated?
2. Are manual keystrokes per PR going down?
3. How much accepted AI code survives to merge?
4. Are cycle times improving?
5. Is quality holding steady?

If those five move in the expected direction, your hypothesis is gaining support. If AI-originated share rises but human keystrokes stay flat and rewrite rates are high, then AI is helping, but not yet displacing hand typing in the strong way your hypothesis claims.
