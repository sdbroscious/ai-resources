## Why Measuring AI vs. Human Code Contribution Is Hard
### A Summary for Technology Leadership

**The core problem: "AI-generated code" is not a single, agreed-upon thing.**
Before you can measure it, you have to define it — and there are at least four defensible definitions:

- AI *suggested* the code (the model produced the tokens)
- AI *inserted* the code (the developer accepted the suggestion into the editor)
- AI *delivered* the code (AI-inserted lines that survived review and were merged)
- AI *influenced* the code (the solution strategy came from AI, even if the developer rewrote it)

GitHub Copilot's metrics are closest to the second definition — lines inserted via acceptance actions — and do not directly represent delivered code, nor do they capture cognitive contribution. Most reporting conflates these definitions, which makes comparisons unreliable.

---

**Technical measurement gaps — what the tools simply cannot see:**

- If a developer accepts a 15-line suggestion and then modifies 10 lines or deletes 5, none of those post-acceptance changes are attributed back to the original AI suggestion.
- Manual copy-paste from Copilot Chat is not counted, and code "inspired by" AI but rewritten by the developer is entirely untracked.
- Specific minimum IDE and plugin versions are required for telemetry; developers on older versions contribute zero LOC data, systematically underreporting usage.
- 59% of developers now use three or more AI tools simultaneously; single-tool metrics capture only a partial picture.

---

**Statistical / methodological pitfalls — the math doesn't hold up cleanly:**

- The most intuitive ratio — Copilot lines added divided by total repository lines changed — mixes two different measurement pipelines: editor-level telemetry in the numerator and committed repository diffs in the denominator.
- This mismatch can cause the ratio to exceed 100%, drop artificially during refactoring weeks, or jump artificially when developers upgrade IDE plugins — none of which reflect actual changes in AI usage.
- Agent mode is categorically different from inline autocompletion: an agent scaffolding an entire feature generates hundreds of lines in a single operation, which can make the contribution rate appear to spike without any meaningful change in how developers are working.

---

**Behavioral / gaming risks — measuring it changes it:**

- When metrics become targets, predictable distortions follow: developers may favor workflows that "count" (built-in accept actions) over equally productive workflows that don't (manual copy-paste), changing the metric without changing actual AI reliance.
- Large agent-driven refactors may be favored to inflate "AI lines," even when smaller, safer architectural changes would be preferable.
- Both documents explicitly flag Goodhart's Law: any AI contribution metric used for performance evaluation or organizational targets is vulnerable to the very behaviors it is meant to measure.

---

**Bottom line for leadership:**
The most defensible posture is to treat any AI code contribution percentage as a *floor estimate and a directional indicator*, not a precise measurement. It answers "how much code comes from AI?" — not "is that code good?" — and should always be paired with at least one quality indicator such as defect rates or code churn.
