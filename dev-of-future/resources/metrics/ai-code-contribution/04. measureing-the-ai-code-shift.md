# Measuring the shift from hand-written to AI-generated code

**Developers are writing less code by hand — but proving it rigorously is far harder than it sounds.** The industry has no standard definition of "AI-generated code," current tooling captures only a fraction of AI's real contribution, and the most commonly cited metrics (acceptance rate, lines of code) systematically mislead when taken at face value. For a ~350-developer organization on GitHub Copilot pursuing an SDD pilot, the most defensible approach combines Copilot's native telemetry as a floor estimate with git-level churn analysis, quality counterweights, and developer self-reports — framed not as a single percentage but as a multi-dimensional trend over time.

The core premise — that hand-written code declines as AI tooling matures — is directionally supported by industry data (GitHub reports **46% of active users' code** now originates from Copilot, up from 27% at launch). But this headline number masks serious measurement complexity: accepted code that's immediately edited, code pasted from ChatGPT that no telemetry sees, and the uncomfortable finding that experienced developers may actually slow down with AI tools. What follows is a practical framework for navigating this measurement challenge.

---

## No one agrees on what "AI-generated" means

The most fundamental obstacle is definitional. There is no industry-standard taxonomy for classifying code authorship along the human-AI spectrum. In practice, code exists on a continuum that current tooling mostly ignores:

**Fully human-written** code (no AI tool active) sits at one end. At the other end lies **fully autonomous agent output**, where tools like Copilot Agent or Claude Code scaffold entire features across multiple files from a single prompt. Between these poles lie several intermediate states: inline completions accepted without modification ("ghost text"), suggestions accepted then heavily edited, code generated via chat and pasted into the editor, and — most invisible — code the developer wrote themselves after being *inspired* by a rejected suggestion.

The operational definitions vary sharply by vendor. **GitHub** defines acceptance rate as "number of acceptances normalized by shown completions" and tracks persistence rate (code unchanged after a time period), but explicitly avoids claiming these measure "AI-generated code" — the documentation frames LOC metrics as a "directional measure of Copilot output." **Windsurf** offers the most granular definition through its Percentage of Code Written (PCW) metric, where any AI text later edited by the developer is reclassified as human-written — a conservative approach that penalizes superficial inflation. **DX** uses file-system-level observability to detect batch modifications characteristic of AI tools, making it tool-agnostic. None of these definitions are interchangeable.

Your own internal analysis correctly identifies four distinct concepts that get conflated: AI-suggested text (what the model proposed), AI-inserted text (what entered the editor), AI-delivered code (what survived to merge), and AI-owned change (where the solution strategy came from AI even if a human typed the final version). Copilot telemetry captures only the first two. No commercial tool reliably measures the fourth.

## The metrics landscape spans volume, quality, and experience

Metrics for tracking the human-to-AI shift fall into three categories, each with distinct strengths and failure modes.

### Volume and adoption metrics

**Acceptance rate** — the percentage of shown suggestions a developer accepts — is GitHub's most visible metric and, per their own peer-reviewed research (Ziegler et al., ACM Communications 2024), the best predictor of *perceived* productivity among all telemetry measures. Industry-wide acceptance rates hover around **27–30%**. However, acceptance rate measures adoption behavior, not code quality or value. A developer may accept a suggestion because rejecting and rewriting costs more cognitive effort than accepting and editing — a "good enough" dynamic that inflates the metric without reflecting genuine productivity.

**Lines of code metrics** from Copilot's GA dashboard (February 2026) track `loc_added_sum`, `loc_suggested_to_add_sum`, `loc_deleted_sum`, and `loc_suggested_to_delete_sum` across completions, chat, and agent features. These are event-level measurements captured at the moment of acceptance — **no post-acceptance tracking exists**. If a developer accepts 15 lines then edits 10, all 15 remain counted as AI-generated. Your internal framework proposes AI Code Contribution Rate = Copilot `loc_added_sum` / GitHub org total lines added, with the critical framing that this is a "floor estimate" that undercounts (missing other tools, chat-based usage) while simultaneously overcounting (including accepted-then-deleted code).

**Windsurf's PCW** (Percentage of Code Written) provides character-level attribution: PCW = W / (W + D), where W is persisted AI-originated bytes and D is human-typed bytes, with edited AI text counting as D. This is methodologically superior to GitHub's approach but locked to Windsurf's ecosystem. With agentic workflows, PCW numbers routinely exceed **90%**, though Windsurf itself notes the actual productivity lift is roughly half the PCW value.

### Quality and sustainability metrics

**Code churn** is the most important counterweight to volume metrics. GitClear's analysis of **211 million changed lines** (2020–2024) found code churn nearly doubled from the pre-AI baseline, AI-generated code showed a **41% higher churn rate** than human-written code, refactoring activity collapsed by 60%, and duplicated code blocks increased eightfold. These findings suggest that rising AI code volume may be partly offset by rising instability.

**PR-level quality metrics** offer more actionable signals. CodeRabbit's analysis of 470 PRs found AI-co-authored pull requests contained **1.7x more total issues**, with logic errors 75% more frequent and security findings 57% higher. Veracode's study of 100+ LLMs found **45% of AI-generated code** contains security flaws, with the vulnerability rate remaining flat regardless of model size — newer models do not generate meaningfully more secure code.

**DORA metrics** provide the broadest organizational lens. The 2024 DORA report (39,000+ professionals) found AI adoption correlates with small improvements in documentation quality (+7.5%) and code review speed (+3.1%), but a **7.2% decrease in delivery stability** and **1.5% decrease in throughput**. The 2025 report shows AI now correlates positively with throughput but continues to hurt stability. AI appears to amplify existing organizational strengths and weaknesses rather than uniformly improving outcomes.

### Experience and perception metrics

**Developer self-reports** from DX's platform indicate average time savings of **2–3 hours per week** across hundreds of organizations — substantially more modest than vendor marketing claims of 55% productivity gains. The Stack Overflow 2025 survey (49,000 developers) reveals a trust crisis: **46% distrust AI output accuracy** (up from 31% in 2024), and 66% cite "almost-right solutions" as their biggest frustration. The JetBrains 2025 survey shows 19% of developers claim saving 8+ hours per week, but only 44% report AI as fully adopted in their workflows.

The most sobering data point comes from the **METR randomized controlled trial** (July 2025): 16 experienced open-source developers working on their own repositories took **19% longer** with AI tools while believing they were 20% faster — a perception-reality gap of nearly 40 percentage points. This is the only known RCT measuring AI impact on experienced developers doing real work on familiar codebases, and it directly challenges the universal-speedup narrative.

## Tools for instrumentation range from free to enterprise-grade

For a Copilot-standardized organization, the tooling decision tree has three tiers.

**Tier 1 (no additional cost)** is GitHub's native Copilot Metrics dashboard and API, now GA. It provides daily active users, acceptance rates, lines of code by feature/language/IDE, and PR lifecycle metrics at enterprise and org levels. User-level data is API-only. The key limitation is Copilot-only visibility — it cannot see code from Cursor, Claude Code, ChatGPT, or any other tool. Microsoft's open-source Copilot Metrics Dashboard (deployable on Azure) and the community Copilot Metrics Viewer both visualize this data. Infrastructure requirements include minimum IDE versions (VS Code v1.102.0+, JetBrains 2024.2.6+) and 2–3 day data lag.

**Tier 2 (enterprise platform)** includes DX, Jellyfish, and Faros AI. **DX** stands out for combining file-system-level AI detection (tool-agnostic), developer experience surveys, and a research-backed three-dimensional framework (Utilization, Impact, Cost). DX's "same-engineer" methodology — comparing each developer against their own pre-AI baseline — is the most statistically defensible approach for isolating AI's effect. **Jellyfish** has the broadest multi-tool coverage (7+ tools including Copilot, Cursor, Claude Code) and the largest benchmark dataset (300+ companies, 20,000+ engineers). **Faros AI** offers the strongest cause-effect analysis and executive reporting capabilities. All three are enterprise SaaS with custom pricing.

**Tier 3 (code-level attribution)** includes newer entrants. **Exceeds AI** performs line-level AI detection across PRs using code pattern analysis, commit message parsing, and optional telemetry, claiming 95%+ accuracy. **BlueOptima** uses ML trained on 16 billion+ code metric observations across 800,000+ engineers. The open-source **Git AI** project takes a fundamentally different approach — provenance-based tracking that records the origin of each edit in real time via `.git/ai/` checkpoints, producing commit-level attribution like "92% AI, 6% you, 2% mixed." This is the most accurate method but requires installation on every developer machine.

## Industry benchmarks reveal a complex, contradictory picture

The headline statistics cited by tech CEOs paint a clear trend. Google's Sundar Pichai reported **>25% of new code** at Google is AI-generated (Q3 2024), rising to >30% by Q1 2025. GitHub reports Copilot contributes **46% of active users' code**. Microsoft's Satya Nadella estimated 20–30% of repo code is AI-written. Y Combinator reported 25% of Winter 2025 startups had codebases that were 95% AI-generated.

These numbers require heavy qualification. GitHub's 46% applies only to *active Copilot users*, not all developers. Google's figure is an earnings-call claim without published methodology. The specific construct being measured (lines? change lists? characters?) is rarely disclosed. Jellyfish data from 300+ companies shows almost half now have at least 50% AI-generated code by end of 2025, but this reflects the vendor's customer base (early adopters), not the industry average.

The controlled studies tell a more nuanced story. The largest field experiment to date — **Cui et al. (2025)**, three RCTs across Microsoft, Accenture, and a Fortune 100 company totaling 4,867 developers — found a pooled **26% increase in completed pull requests**, with less experienced developers showing the greatest gains. But individual experiments were noisy and often not statistically significant alone. The studies used GPT-3.5-era Copilot, making direct extrapolation to current models uncertain.

The critical tension: lab experiments with well-scoped tasks consistently show **21–56% speed improvements**. Real-world measurements on complex, brownfield codebases show **flat to negative** effects for experienced developers, with DX finding a more realistic **2–3 hours saved per week** (~5–10% of a work week). The gap between lab conditions and production reality remains the central unresolved question in the field.

## Five confounding factors that undermine simple measurement

**The denominator mismatch problem** is the most technical pitfall. Computing "AI share" by dividing Copilot `loc_added_sum` (editor-level telemetry) by git additions (committed repository diffs) combines two fundamentally different measurement systems at different lifecycle stages. This can produce ratios exceeding 100% when Copilot-generated edits occur in scratch branches later rebased, and artificial drops during reformat weeks. Your internal analysis correctly identifies this and recommends treating the two series as parallel trend lines rather than a single ratio.

**The multi-tool blind spot** is worsening. The Stack Overflow 2025 survey shows **59% of developers use three or more AI tools** simultaneously. Copilot telemetry captures only its own contributions. Code pasted from ChatGPT, suggestions from Cursor, or output from Claude Code are invisible to GitHub's dashboard. Any single-tool metric is structurally incomplete and will become more so over time.

**Goodhart's Law operates at multiple levels** in this domain. Lines of code was already a discredited metric before AI — Dijkstra called them "lines spent," Bill Gates compared LOC to measuring airplanes by weight. AI makes the metric infinitely gameable because the cost of generating a line of code is now functionally zero. An agent that scaffolds an entire feature generates hundreds of lines in a single operation, categorically different from an inline autocompletion that saves a developer from typing `for (int i = 0;`. Reporting both as "AI-generated lines" conflates fundamentally different activities.

**The "autocomplete illusion"** refers to AI completing code the developer would have typed anyway. When suggestions align with developer intent, the developer experiences a feeling of speed without actual net gain. This is inherently unmeasurable — no telemetry can determine counterfactual typing speed.

**Security and quality degradation** is the most operationally consequential concern for an insurance carrier. Veracode found AI-generated code has **2.74x more vulnerabilities** than human-written code, with cross-site scripting at 86% failure rate and log injection at 88%. A **40% increase in secrets exposure** (hardcoded API keys, passwords) has been documented in AI-generated projects, because models learn from training data where developers committed credentials. For a regulated financial services organization, this risk profile demands that any volume metric be paired with security scanning data.

## A recommended metric framework for leadership and SDD validation

The framework below is designed to provide defensible answers to two questions: "How much of our code is AI-generated?" (leadership reporting) and "Is SDD producing better outcomes than traditional AI-assisted development?" (pilot validation). It builds directly on the AI Code Contribution Rate framework and SDD Governance Plan already in your Google Drive.

### For ongoing leadership reporting: the three-panel dashboard

**Panel 1 — AI Code Contribution Rate (trend line).** Copilot `loc_added_sum` / GitHub org total lines added, reported monthly with quarterly trend analysis. Apply the normalization decisions already documented: scope to application code repos, filter to source code file types, exclude bot/CI commits, and acknowledge the population alignment gap. Frame explicitly as a floor estimate. This answers the "how much" question with appropriate caveats.

**Panel 2 — Quality counterweight.** Pair the volume trend with at least two quality signals: code churn rate (lines reverted or substantially modified within 14 days of merge, segmented by AI-heavy vs. AI-light repos) and change failure rate from DORA metrics. If AI contribution rises while churn stays flat or declines, that's a genuinely positive signal. If both rise together, the volume gain is partly illusory.

**Panel 3 — Developer experience.** Quarterly pulse survey measuring self-reported time savings (hours/week), trust in AI output, and workflow satisfaction. DX's finding that organizations average 2–3 hours saved per week provides a benchmark. Track the percentage of developers reporting >4 hours saved as your "power user" cohort — growth in this segment is a leading indicator.

### For SDD pilot validation: outcome-based metrics

The SDD pilot should be measured on a fundamentally different axis than volume. The relevant question shifts from "how much code did AI write?" to "does AI-generated code satisfy the specification?" This naturally resists Goodhart's Law because specifications encode business intent, not code volume.

- **Cycle time from spec start to merge to release** — the primary velocity metric, tracked per spec ID using the traceability system in your governance plan
- **Defect escape rate** — defects that pass spec tests and reach production, the primary quality metric
- **Spec/test churn after approval** — a leading indicator of specification quality; high churn suggests specs aren't capturing requirements well enough for AI (or humans) to implement correctly
- **Time-to-add-scenario** — how quickly can a team add a new scenario and passing test? Your exit criterion of <1 day is appropriate
- **AI code contribution rate within SDD repos vs. non-SDD repos** — this is where the volume metric becomes genuinely informative, because the SDD framework provides the quality assurance that makes higher AI contribution rates a positive signal rather than a risk

### The comparison that proves the premise

To test whether developers produce less hand-written code over time, you need a **same-team longitudinal comparison**: measure each team's AI Code Contribution Rate monthly and plot the trajectory. The slope of this line — not any single-month percentage — is the answer to the leadership question. Pair this with DX's "same-engineer" methodology if you adopt their platform, which compares each developer against their own pre-AI baseline to eliminate confounding variables.

For the SDD pilot specifically, the strongest evidence would be a **controlled comparison**: SDD teams vs. non-SDD teams using the same Copilot tooling, measuring whether SDD produces higher AI code contribution rates *and* lower defect escape rates simultaneously. If SDD teams show both higher AI utilization and better outcomes, that's a defensible case that structured specifications enable more effective AI code generation — not just more volume.

### What to present and what to caveat

Present the AI Code Contribution Rate as a trajectory, not a point estimate. Note that it is a floor estimate (missing other AI tools and chat-based generation) that will jump discontinuously as agent-mode adoption increases — document this now so leadership isn't surprised. Always pair it with quality data. Never report it at the individual developer level. And acknowledge openly that the industry has not settled on a standard measurement approach: even GitHub calls its own LOC metrics "directional." The organizations that will have the most credible data in 12 months are the ones that document their methodology, normalization choices, and known blind spots today.

---

## Conclusion

The premise that hand-written code declines over time as AI tooling matures is almost certainly true directionally — adoption data from GitHub, Google, and industry surveys all point the same way. But the magnitude, pace, and implications of this shift are far less certain than vendor marketing suggests. The most rigorous independent research (METR, DORA, GitClear) consistently shows smaller, more ambiguous effects than controlled lab studies, and raises legitimate concerns about code quality, security, and delivery stability.

For a commercial insurance carrier, where code correctness has regulatory and financial consequences, the right measurement approach is deliberately conservative: treat volume metrics as adoption telemetry rather than productivity proof, always pair them with quality and stability counterweights, and use the SDD pilot as a controlled experiment that demonstrates AI can produce *better* outcomes — not just *more* code. The shift from hand-written to AI-generated code is real and accelerating. The harder and more valuable question is whether that shift is making your software better, and that requires metrics that go well beyond counting lines.
